SalesDB Sample Application

 

The SalesDB application is an example of a simple MobiLink enabled application.  Before you can use it, you must create two SQL Anywhere databases:  SalesDB and SalesDB_remote.  This is done by running this command file:
           

%Code Examples%\MobiLink\SalesDB\db\makedb.cmd

 

%Code Examples% represents your PowerBuilder Code Examples folder.

 

If SQL Anywhere 11 is installed on your workstation, makedb.cmd will create SQL Anywhere 11 databases.  Otherwise, it will attempt to create ASA 10.x databases.     

 

This SalesDB application differs slightly from the sample application you may remember from PocketBuilder or ASA 9.  First, it makes SalesDB_remote a strongly encrypted database (dbkey=secret).  Second, the dba.SyncSettings table is automatically added to the database schema in SalesDB_remote.  Third, the ODBC datasource for SalesDB_remote no longer has UID and PWD specified in the connectstring, since this compromises database security.

 

Once the databases are created, run the following command file to start the synchronization server:

%Code Examples%\MobiLink\SalesDB\db\StartML.cmd

 

Next, run PB IDE and open the MobiLink sample workspace:

%Code Examples%\MobiLink\MLSamples.pbw

 

The MobiLink Server Version property in nvo_salesdb_mlsync.sru is currently set to 11.  If salesdb_remote.db and salesdb.db were created as ASA 10 databases, then you must manually edit the nvo_salesdb_mlsync.sru object in salesdb.pbl.  Go to the “Settings” tab in the properties view, change the MobiLink Server Version property to 10, and save the changes.

 

When you run the SalesDB target, you will notice the SmartTool object, w_login_prompt, 

asking for db password (sql) and encryption key (secret).  This provides much

better security than hard-coding this info in the ODBC datasource!

Once you connect to the database, the application detects a need to synchronize the

remote database initially, so the w_salesdb_sync_options window is opened.   

The first time through, you'll need to fill in a lot of missing properties:

            mluser = 50

            mlpass = “”

            dbuser = dba

            dbpass = sql

            encrypt = secret

Then click OK.  This launches the first synchronization which will download a lot of data to the remote database.  Subsequent requests will only download data that has changed on the consolidated database since the last synchronization was run.

 

Next, choose "File, Sync Options...".  Notice that now MLUser, DBuser, DBPass, and

encryption key fields are already filled in for you!  All you have to do is click

OK and sync occurs again.

 

The “File, Sychronize” menu item also performs synchronization, but it does not open  the w_salesdb_configure_sync window.  Rather, it uses the synchronization properties from the previous successful synchronization.  The gf_salesdb_sync function was edited slightly to provide this functionality:

//  This gets the syncparm settings from the previous sync attempt

            gf_get_syncsettings_from_db(sqlca, l_uosync.syncregistrykey, parms)

            l_uosync.SetParm(parms) 

 

The "Change DB Password" and "Change ML Password" menu items illustrate the use of additional objects from the SmartTool library.  Try them and notice that the once you change the DB password or ML password, the change is also saved to the dba.SyncSettings table so that future synchronization requests are properly initialized with the new password.
